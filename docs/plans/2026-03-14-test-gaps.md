# Implementation Plan: Test Gaps (FLI-86, FLI-87, FLI-90, FLI-91)

## Issues
- FLI-86: test: use mix_stderr=False in search_fast error path test for precise stderr assertion
- FLI-87: test: add non-default time test for _make_snapshot parameterization
- FLI-90: Add test asserting parse_date calendar-vs-format error message distinction
- FLI-91: Add test for non-numeric cb_backoff fallback branch in health.py

## Research Context

All four issues are test gap improvements identified during code review of PRs #35 and #36. All changes go into `tests/test_cli.py`.

### Codebase patterns
- Tests use `typer.testing.CliRunner` with default `mix_stderr=True` everywhere
- Test classes: `TestSearchCommands`, `TestHealthCommand`, `TestReport`
- Helper factories are inline methods on test classes (e.g., `_make_snapshot`, `_make_urlopen_mock`)
- Assertions always include output on failure: `assert result.exit_code == 0, result.output`

### Source code under test
- **search.py** `_print_search_error()` (lines 17-27): calls `typer.echo(..., err=True)` twice — category line and hint line go to stderr
- **validators.py** `parse_date()` (lines 20-26): two error paths — regex mismatch → "expected format YYYY-MM-DD", valid format but invalid date → "not a valid calendar date"
- **health.py** `health_check()` (lines 56-60): `float(cb_backoff)` in try block, `except (ValueError, TypeError)` prints raw value

## Decisions Made

1. **FLI-86:** Update existing `test_search_fast_error_displays_category_and_hint` to use `CliRunner(mix_stderr=False)` and assert on `result.stderr` instead of `result.output`. Also update the similar `test_search_error_displays_category_and_hint` test for consistency (both test the same `_print_search_error` function).
2. **FLI-87:** Add `test_report_show_with_custom_times` in `TestReport` that calls `_make_snapshot` with non-default departure/arrival times and verifies the rendered output contains the custom times.
3. **FLI-90:** Add `test_search_fast_rejects_invalid_calendar_date` (and optionally `test_search_latam_rejects_invalid_calendar_date`) that passes `2026-02-30` and asserts "valid calendar date" is in the error message, distinct from "format YYYY-MM-DD".
4. **FLI-91:** Add `test_health_cb_backoff_non_numeric_string` in `TestHealthCommand` that sets `backoff_remaining_sec` to `"not_a_number"` and asserts the raw value is printed (exercises the `except (ValueError, TypeError)` branch with a truly non-numeric string).
