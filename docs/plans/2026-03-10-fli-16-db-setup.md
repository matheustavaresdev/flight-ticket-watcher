# Implementation Plan: SQLAlchemy + Alembic Setup

## Issues
- FLI-16: SQLAlchemy + Alembic setup

## Research Context

**Project stack:** Python 3.12, Hatchling build, src-layout (`src/flight_watcher/`).

**Docker:** PostgreSQL 18-alpine via `docker-compose.yml` with health checks. Scanner service depends on healthy DB. Env vars already defined in `.env.example`: `POSTGRES_HOST=db`, `POSTGRES_PORT=5432`, `POSTGRES_DB=flight_watcher`, `POSTGRES_USER=flight_watcher`, `POSTGRES_PASSWORD=changeme`.

**Existing patterns:** Env vars accessed via `os.environ.get()` with defaults at module level. Logging via `logging.getLogger(__name__)`. No dedicated config module exists yet.

**Current models:** `src/flight_watcher/models.py` has a `@dataclass FlightResult` — converting to ORM is out of scope for this issue (that's FLI-13's territory).

**Test patterns:** pytest with `unittest.mock.patch()`. Test config in `pyproject.toml` sets `testpaths = ["tests"]` and `pythonpath = ["src"]`.

## Decisions Made

- **PostgreSQL driver:** `psycopg[binary]>=3.1` — psycopg3 is the modern async-capable driver, matches the project's preference for current tooling. Binary variant avoids needing `libpq-dev` in Docker.
- **SQLAlchemy style:** 2.0 API with `DeclarativeBase` (not legacy `declarative_base()`).
- **Database module:** Single file `src/flight_watcher/db.py` — the project is small, a `db/` package is premature. Contains engine factory, session factory, and `Base` class.
- **Connection string:** Built from individual `POSTGRES_*` env vars (already in `.env.example`). Also support `DATABASE_URL` override for flexibility.
- **Alembic location:** `alembic/` at project root (standard convention). `alembic.ini` at project root.
- **Alembic env.py:** Reads DB URL from the same env vars as the app (via `db.py`), no hardcoded URLs in `alembic.ini`.
- **No model conversion yet:** `models.py` dataclass stays as-is. `db.py` exports `Base` for future ORM models (FLI-13).
- **Dockerfile update:** Add `alembic upgrade head` before app start so migrations run on container boot.

## Implementation Tasks

1. **Add dependencies to `pyproject.toml`** — affects `pyproject.toml`
   - Add `sqlalchemy>=2.0`, `alembic>=1.13`, `psycopg[binary]>=3.1` to `dependencies`

2. **Create database module** — affects `src/flight_watcher/db.py` (NEW)
   - Build `DATABASE_URL` from env vars (with `DATABASE_URL` env override)
   - Create `engine` factory function: `create_engine(url)` with sensible defaults (`pool_pre_ping=True`)
   - Create `SessionLocal` factory via `sessionmaker`
   - Export `Base = DeclarativeBase` subclass for ORM models
   - Export a `get_session()` context manager for clean session lifecycle

3. **Initialize Alembic** — affects `alembic/` directory, `alembic.ini`
   - Run `alembic init alembic` to scaffold
   - Edit `alembic.ini`: remove hardcoded `sqlalchemy.url` (will be set programmatically)
   - Edit `alembic/env.py`:
     - Import `Base` from `flight_watcher.db` and set `target_metadata = Base.metadata`
     - Import DB URL builder from `flight_watcher.db` for `run_migrations_online()`
     - Use `config.set_main_option("sqlalchemy.url", url)` in both online and offline modes

4. **Update Dockerfile entrypoint** — affects `Dockerfile`
   - Change entrypoint to a shell script or inline command that runs `alembic upgrade head` then `python -m flight_watcher`

5. **Write connection verification test** — affects `tests/test_db.py` (NEW)
   - Test that `get_database_url()` builds correct URL from env vars
   - Test that `DATABASE_URL` env override takes precedence
   - Test that `get_session()` context manager yields a session and closes it
   - Mock env vars, do NOT require a running database

6. **Verify Docker connectivity** — manual verification step
   - `docker compose up -d db` → wait for healthy
   - `docker compose run --rm scanner alembic current` → should connect and show no migrations
   - `docker compose down`

## Acceptance Criteria
- SQLAlchemy, Alembic, and psycopg are in `pyproject.toml` dependencies
- `src/flight_watcher/db.py` provides engine, session factory, and Base
- `alembic/` directory exists with properly configured `env.py`
- `alembic.ini` does NOT contain hardcoded credentials
- Database connection works inside Docker network (`scanner` → `db`)
- Dockerfile runs migrations on startup before the scanner
